# 论文精读

四步工作流：定位 → 提取 → 结构化总结 → 交互问答。

## 第一步：在 Zotero 中定位论文

根据用户提供的信息，用 `zot` CLI 定位目标论文。

**如果用户提供了论文标题关键词：**
```bash
zot search "<关键词>" --limit 5
```

**如果用户想找最近添加的论文：**
```bash
zot recent --limit 10
```

**如果用户指定了 Zotero 分类（collection）：**
先用 `zot collection list` 查看可用分类，再用 `--collection` 过滤：
```bash
zot search "<关键词>" --collection "<分类名>"
```

将搜索结果展示给用户确认。如果结果不唯一，让用户选择要读哪一篇。

## 第二步：提取论文内容

确认目标论文后（记下 Item Key，如 `ABC123`），分两步获取信息：

### 2.1 先读元数据
```bash
zot --detail full read <KEY>
```
这一步获取标题、作者、出版年份、会议/期刊、摘要等结构化信息。

### 2.2 再提取全文
```bash
zot pdf <KEY>
```
`zot pdf` 会利用本地缓存，重复读取很快。如果只想看特定部分：
- `zot pdf <KEY> --outline` — 先看章节目录
- `zot pdf <KEY> --pages 1-5` — 按页码范围提取
- `zot pdf <KEY> --section 3` — 按章节号提取

如果用户特别关心论文中的标注/高亮：
```bash
zot pdf <KEY> --annotations
```

提取的文本用 Read 工具读入。如果文本很长，优先以 `--outline` 了解结构，再有针对性地按章节读取。不要一次性全量读入导致上下文溢出。

### 提取质量检查

如果 `zot pdf` 提取的文字乱码或质量差（常见于扫描版论文），尝试：
```bash
zot pdf <KEY> --extractor pymupdf
```
或：
```bash
zot pdf <KEY> --extractor mineru
```

## 第三步：八段式结构化总结

读完论文内容后，输出完整的中文总结。

### 输出方式

总结分为两部分产出：

1. **Markdown 文件**通过 **obsidian-markdown** skill 写入 Obsidian Vault，确保格式兼容 Obsidian 的 Callout、Dataview、Mermaid 等特性。术语表和核心引用表使用 **obsidian-bases** skill 的数据表格式（`.md` 文件中的 `dataview` 兼容表格）。
2. **逻辑流画板**通过 **json-canvas** skill 生成 `.canvas` 文件，在 Obsidian Canvas 中打开即可看到论文逻辑骨架的图形化视图。

### Markdown 模板

文件命名：`<论文中文译名>.md`

```markdown

> [!info]- 论文元数据
> | 属性 | 内容 |
> |------|------|
> | **英文标题** | |
> | **作者** | |
> | **年份/会议/期刊** | |
> | **Zotero Key** | |
> | **DOI** | |

### 2.3 本文切入角度

### 5.2 主要对比
```

### 逻辑流 Canvas（json-canvas）

总结 Markdown 写完后，调用 **json-canvas** skill 生成论文逻辑流 Canvas 画板：

- **节点**：每个逻辑步骤一个节点（如「痛点：语义不可解释」「对策：三元组表示」「模型：TESC 系统」「实验：情感分析 + QA」）
- **边**：标注因果关系（`→` 推导/继承，`⇢` 数据流）
- **颜色**：痛点用红色系、对策用绿色系、方法用蓝色系、实验用黄色系
- **文件命名**：`<论文英文标题简写>_logic.canvas`

如果论文有复杂系统架构（收发机框图、信号流、帧结构等），额外生成一个技术架构 Canvas：`<论文英文标题简写>_arch.canvas`。

### 输出原则

- **使用 Obsidian 兼容格式**：Callout 块（`> [!info]`、`> [!summary]`、`> [!important]`）、dataview 兼容表格。
- **层级分明的编号**：用 `1` `1.1` `1.2` 这样的数字层级代替中文数字，确保在 Obsidian 大纲视图中可以正确折叠和跳转。
- **公式的每个符号必须有解释**。通信论文公式密集、符号多（信道矩阵 H、噪声方差 σ²、调制阶数 M...），每个符号第一次出现时必须说明其物理含义和单位。不能假设读者认识所有符号。直觉解释要覆盖整个公式的物理意义，而非逐项翻译。
- **性能数字要精确**，回到提取文本中核对，不能凭印象。
- **区分作者声称和你的判断**。"作者声称的贡献"放 6.1，「你判断的局限/适用边界」放 5.4。
- **术语表必须完整**。通信论文缩写多（MIMO、OFDM、NOMA、IRS、SINR...），每遇到一个新缩写就加入术语表。
- **如果某节信息缺失**（比如没有 ablation study），写「本文未涉及」，不编造。
- **核心引用文献的选择标准**：不是引用次数最多的，而是理解本文最需要先读的奠基性工作或最相关的竞争方法。

### 可视化（json-canvas）

所有可视化全部通过 **json-canvas** skill 生成 Obsidian Canvas 画板：

**必须生成（1 个）**：
- **论文逻辑流 Canvas**：节点为逻辑步骤（痛点→对策→模型→方法→实验→结论），边标注因果关系，颜色区分模块。文件名：`<论文简写>_logic.canvas`

**按需生成（1 个）**：
- **技术架构 Canvas**：当论文有系统框图、收发机架构、信号流、帧结构、协议栈等复杂技术结构时生成。文件名：`<论文简写>_arch.canvas`

Markdown 总结中不嵌入 ASCII 流程图——图形化内容全部交给 Canvas，Markdown 只放文字和表格。

## 第四步：交互问答

输出总结后，主动询问用户：

> 以上是这篇论文的结构化总结。你是否想深入了解某个方面？例如：
> - 某个公式的详细推导或物理含义
> - 与某篇相关论文的详细对比
> - 方法在你自己的场景中是否适用
> - 复现这篇论文的可行性评估
> - 画一张系统架构的 Canvas 图

根据用户的追问，回到 `zot pdf` 提取的文本中查找相关信息，精准回答。

## 多篇论文处理

当用户同时涉及多篇论文时（如"对比 A 和 B"、"最近 3 篇相关工作的区别"），先确认清楚用户的具体意图（是逐一精读、横向对比、还是只看差异点），再决定处理方式。不要假设用户想要什么格式。
